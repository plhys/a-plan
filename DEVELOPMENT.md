## 🚨 维护者血泪踩坑警示录 (后来者必读)

作为 OpenClaw-Ghost-Agent，我在 4.2.1 到 4.2.3 的演进中踩到了以下三个足以让 Pod 瘫痪的深坑，请后来者引以为戒：

### 1. 幽灵模式的“裸奔”陷阱 (The Ghost Security Gap)
- **坑位**: 追求极速启动（跳过插件加载）时，如果不细化逻辑，会连同鉴权插件 (`default-auth`) 一起关掉。
- **后果**: 服务启动只需 1 秒，但处于全网“裸奔”状态，任何人都能控制你的后台。
- **警戒**: **核心插件（如 Auth）必须在任何模式下被强制加载。** 极致启动不能以牺牲安全为代价。

### 2. 静态 Import 的“连体婴儿”病 (The Static Import Death)
- **坑位**: 在核心服务 (`service-manager.js`) 中使用静态 `import` 引用 Clash 等可选模块。
- **后果**: 这种写法将模块“焊死”在了核心上。一旦用户物理删除了模块文件夹，整个 API 服务会报 `Module Not Found` 并直接崩溃自杀。
- **警戒**: **永远使用 `dynamic import()` + `fs.exists` 探测。** 核心应该是寄主，功能应该是挂件。挂件掉了，寄主必须毫发无伤。

### 3. UI 与后端的“感知断层” (The Feedback Vacuum)
- **坑位**: 后端成功抓到了 52 个节点，但 UI 没写自动刷新逻辑。
- **后果**: 用户点完保存，看到的是一片空白，会产生“没生效”的严重挫败感。
- **警戒**: **极客需要即时反馈。** 所有的长耗时任务（如测速、订阅更新）必须配备前端自动轮询和变色灯反馈。

### 4. 版本号的“尊严” (The Version Asset)
- **坑位**: 为了对齐 UI，随意在本地修改版本文件。
- **结论**: UI 只认 Git Tag。在本地改一万次版本号，也不如在 GitHub 顺延打一个真实的 Tag。
- **红线**: **代码推送到哪里，版本就在哪里。** 严禁手动修改本地 `VERSION` 企图欺骗自动化逻辑。
