## 🚨 绝密：维护者踩坑警示录 (后来者必读)

### 坑位 1：本地版本号幻象 (The Local Version Mirage)
*   **现象**：我试图通过 `fs.edit` 修改本地的 `VERSION` 和 `package.json` 来让 UI 显示新版本。
*   **后果**：被 UI 狠狠打脸。UI 右侧的“新版本”下拉框是直接勾连 GitHub Tags 的。由于远程没有对应的 Tag，UI 会判定本地版本非法并强行引导用户“降级”到旧版。
*   **警戒**：**绝对不要在 Pod 本地手动改版本号！** 想要升版，必须通过 Git Token 直接向 GitHub 推送新的 Tag。代码推送到哪里，版本就在哪里。

### 坑位 2：进程重启执念 (The Restart Fetish)
*   **现象**：我以为修改版本号后必须物理重启 `master` 进程才能让 UI 刷新。
*   **真相**：本项目采用了动态读取架构。`system-api.js` 是实时读磁盘文件的。只要 Git 链路通了，文件变了，刷新浏览器即可。
*   **警戒**：在 Pod 环境里，尽量避免非必要的物理重启。利用项目原生的“一键更新”逻辑，那才是最优雅的热重载。

### 坑位 3：手握金钥匙却在撬锁 (The Token Neglect)
*   **现象**：用户已经提供了 `ghp_***` 远程推送权限，我却在本地折腾了半天。
*   **教训**：作为一个极客代理，要学会利用最高权限从源头（GitHub）解决问题，而不是在寄生环境中打补丁。
